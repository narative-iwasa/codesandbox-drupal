# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

This is a **setup/scaffolding repo**, not a committed Drupal application. Its job is to spin
up a Drupal 11 site in Docker (for CodeSandbox) wired for development with Claude Code. The
Drupal codebase (core, contrib, vendor, custom code) is generated at runtime and is **not
tracked in git**.

Gitignored and generated by `setup.sh` (do not expect these in version control; `setup.sh`
overwrites `composer.json`/`composer.lock` on every run): `web/`, `vendor/`,
`composer.json`, `composer.lock`, `config/`, `mysql/`, and Drupal runtime files
(`web/sites/*/settings.php`, `services*.yml`, `files/`).

Tracked source is the scaffolding: `setup.sh`, `claude-cli-install.sh`, `Dockerfile`,
`docker-compose.yml`, `.devcontainer/`, `.codesandbox/`, `README.md`.

## First-time setup

1. Set `ANTHROPIC_API_KEY` in `.env` (README says `cp .env.sample .env` first, but no
   `.env.sample` exists — create/edit `.env` directly), then `source .env`.
2. `./setup.sh` — extracts Drupal files from the `drupal:latest` image, writes
   `composer.json`/`composer.lock`, runs `docker compose up -d`, installs Drush + dev deps
   inside the container, and runs `drush site:install standard`.

After setup: site at **http://localhost:8080**, admin login **admin / admin**. MariaDB on
`mysql:3306`, database `drupal`, root password `password`.

## Running commands (everything happens inside the container)

Composer, Drush, and PHPUnit run **inside the `drupal` container**, never on the host.
Composer commands use `--working-dir=/opt/drupal`.

- Start / stop: `docker compose up -d` / `docker compose down`
- Shell into the app: `docker compose exec drupal bash`
- Drush: `docker compose exec drupal /opt/drupal/vendor/bin/drush <cmd>`
  (e.g. `... drush cr` to rebuild cache, `... drush cim` / `drush cex` for config)
- Composer: `docker compose exec drupal composer <cmd> --working-dir=/opt/drupal`
- Tests (PHPUnit via `drupal/core-dev`; no repo-level `phpunit.xml`, use core's config):
  - All core-configured tests:
    `docker compose exec drupal /opt/drupal/vendor/bin/phpunit -c web/core`
  - A single test file / method:
    `docker compose exec drupal /opt/drupal/vendor/bin/phpunit -c web/core path/to/Test.php`
    (`--filter testMethodName` to narrow to one method)

## Where custom code goes

Composer installer-paths place custom code at `web/modules/custom/{name}`,
`web/themes/custom/{name}`, `web/profiles/custom/{name}` (contrib goes under the matching
`contrib/` dirs). None exist yet. Because `web/themes`, `web/modules`, `web/sites`, and
`config/` are bind-mounted into the container, edits on the host are live inside it.

## Notes

- No CI is configured, and there are no committed `phpcs.xml`/`phpstan.neon` configs —
  linting relies on the standards bundled by `drupal/core-dev`, run manually in the container.
- `setup.sh` also installs the GitHub CLI (`gh`) on the host and, via
  `claude-cli-install.sh`, the Claude Code CLI, UV, and GitHub Spec-Kit. That script writes
  `~/.claude/settings.json` with `dangerouslySkipPermissions: true`.
